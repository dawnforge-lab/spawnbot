# Spawnbot System Prompt — Architecture

This document describes how the system prompt is assembled and what each layer contributes.

## Assembly Order

The system prompt is built in `src/session/llm.ts:73-100` as a 2-part array:

```
system[0] = [                              <- CACHED by provider (stable across turns)
  Layer 1: Provider Prompt                 <- 1-line intro (same for all models)
  Layer 2: SOUL.md                         <- Operating instructions + identity (primary agents only)
  Layer 3: Agent Prompt                    <- Role specialization (debug, explore, coder, etc.)
  Layer 4: Environment Block               <- Working dir, platform, docs references
].join("\n")

system[1] = [                              <- DYNAMIC (changes each turn)
  Memory Context                           <- Relevant memories from FTS5/vector search
]
```

## Layer 1: Provider Prompt

A single line, identical for all models (`src/session/system.ts`):

```
You are an autonomous AI agent. Your operating instructions and identity are defined in your SOUL below. Follow them.
```

## Layer 2: SOUL.md

The main instruction document. Contains both operational instructions and agent identity.
Source: `.spawnbot/SOUL.md` (falls back to built-in default in `src/soul/index.ts`).

Structure:
```markdown
# Spawnbot                    <- Agent name

## Communication              <- How to talk
## Tools                      <- Tool usage policy
## Working with code          <- Code conventions
## File operations            <- File handling
## Git                        <- Git rules
## Safety                     <- Safety rules
## Code references            <- Reference format
## System                     <- System notes

---

# Identity                    <- Populated by /setup or manually
```

The operational sections above `---` are universal. The identity section below is customized per agent. `/setup` only modifies the identity section.

Users can edit the entire file to customize any aspect of the agent's behavior.

## Layer 3: Agent Prompt

Role-specific prompts from `src/agent/prompt/`. Only included when the agent has a specialized role:
- `debug.txt` — debugging methodology
- `explore.txt` — codebase exploration strategy
- `coder.txt` — coding subagent focus
- `compaction.txt` — context compaction
- `ask.txt`, `summary.txt`, `title.txt`, `orchestrator.txt`

## Layer 4: Environment Block

Generated by `SystemPrompt.environment()`:
```
You are powered by the model named {model}. The exact model ID is {provider}/{model}
<env>
  Working directory: /path/to/project
  Is directory a git repo: yes
  Platform: linux
</env>
You have knowledge files you can read and update:
- /path/.spawnbot/USER.md (about your user)
- /path/.spawnbot/GOALS.md (current objectives)
...
```

## Layer 5: Memory Context (Dynamic)

Built by `buildMemoryContext()` in `src/memory/context.ts`. Token-budgeted injection of relevant memories based on the user's current message. Changes every turn.

## Key Design Decisions

- **One prompt for all models.** No more provider-specific switching. The model's capabilities handle the rest.
- **SOUL.md is user-editable.** All instructions live in a single file the user controls.
- **`---` separator.** Operational instructions above, identity below. /setup only touches identity.
- **system[0] is cached.** Provider prompt caching keeps costs down. Memory is in system[1].
- **Subagents skip SOUL.md.** Only primary/user-facing agents get personality. Subagents have their own agent prompts.
